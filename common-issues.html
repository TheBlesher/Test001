Common PLC/WinMOD issues

In the automotive industry most PLC logic files are made in Rockwell’s Software, Studio 5000.



The programming language is known as Ladder Diagram or ladder logic, when reading it, we analyze it from top to bottom.
In WinMOD we create all the physical I/O devices that send and receive data from and to the PLC. In other words, if there is a device the PLC needs to communicate with, we will create it on WinMOD.
On the PLC side we will find these devices at the bottom of the Controller Organizer, and on WinMOD those are the elements that we need to add from our components library.


• Controller Organizer
IMPORTANT, it allows to access everything that happens in the Cell. Every folder and branch can be accessed from this panel.


• Tags
A tag is a name you assign to an address of a device/PLC. There are two general scopes for tags, global and local tags. Global refers to tags that can be accessible from every task and routine in the PLC, they can be found in “Controller Tags”.


“Enter Name Filter…”
Gives the ability to search for any Global Tag


Local tags refer to tags that are only available in one task and its routines. Under every routine we find Parameters and Local Tags, this takes us to that station’s local tags.


• “Tasks” folder
This folder gives us access to every task and its routines. In GM standard every station, robot and HMI will have its own task, inside each one, we can find all the connections and processes that take place in that specific fixture or item.


Something similar as in WinMOD when we open the Project Window (F4).


• SafetyTask
Safety tasks are more important than normal ones, that’s why they are separated and will execute every set amount of time regardless of a normal task in progress.


• Accessing a Station “CN 160”
If we want to work in a specific Station, we need to go to the “Tasks” folder and click on the station name. This will open a long list with all the items, sequences, and actions, which take place in it.


In here we find all the logic for that station, most of it is standard and most of the time will look the same.


The main difference should come into the station’s sequence.

Part Presents

• The branches that we see under “S000_Sequence” are the flow that takes place in the Cell. This would be the sequence that we see in the HMI, when we work in the “Scroll List”


• The part present logic is found in “S030_RearCompSpiderSubASMPres” as this branch is located right after the Drop of CN150R04, which helps us to interpret that once the Robot makes the drop the Part Presents will be activate on the Station.


Fundamental Instructions working with the Logic.
• OTE ---> Output Energize, it executes an action, however, it only takes place when all the previous conditions are true.


• XIO ---> Examine if Open, this examines if the contact is OPENED. It is the opposite to the XIC.


• XIC ---> Examine if Closed, this examines if the contact is CLOSED.


• Looking at the Hardwire Prints, as well as in the WinMOD file, we see that this station has 3-part presents. These are connected to Block 01 “CN160BK01”. Each one connected to a different Input.
PP1 ---> .I_02
PP2 ---> .I_03
PP3 ---> .I_04


• In the logic, we can see the rung where the mapping of the Part Presents inputs to the device takes place. For example, BK01.ID02 and PPX01.Sts, if the connections are made successfully on WinMOD, once the SIM is active “running” the Input (XIC) will be green (on), for the Status to take place (energize) as well. This is helpful to know because this way you can tell is a connection is wrong on WinMOD or if a tag is misplaced on the PLC.


• This is a good example to see the difference between local and global tags, the grey tag “CN160BK01.I.D02” is a global tag, this means it can be accessed from anywhere, not only in this specific station, while BK01.I.D02 is a local tag that is aliasing to “CN160BK01.I.D02”. This means that BK01.I.D02 if used inside CN160 will point to CN160BK01.I.D02, but if we use it in CN150 it will point to CN150BK01.I.D02. Aliasing means that it will have a “name” locally, but it will be called different globally. Making it so the same name can be used in different stations, because they will share the same name locally but will be aliased to a different name globally.


• On rung 3 of the PP logic, when the condition of Communication OK” is met, we get three possible paths.
1. When all three (PP1, PP2 AND PP3) status are ON and the station’s style is 12, the “Part Present” will take place as it will be energized.
2. When PP1 and PP2 status are ON, but PP3 status is NOT ON and the station’s style is 52, the “Part Present” will take place.
3. Lastly, when all three (PP1, PP2 AND PP3) status are NOT ON, the “Part Not Present” will take place.


• Parts can also be simulated in tryout mode, to do this we use a part present memory.
• This rung shows us that when the Station is in try out mode “Mode.Tryout” (the station will run without parts), Load Position “LoadPosn” (the station is ready to receive a part), robot R04 it’s in drop 1 “CN150R04.Seg[30]” and also at End of Segment “EndOfSeg” and the Part Not Present “NotPresSwitches” is on (meaning the station has no parts). The Part Present Memory “PresMemory” will be latched (meaning that the output will be maintained) if every previous condition was met.


• Then, this memory will be unlatched in two scenarios, when we are not in Tryout or when we are in tryout, and the robot that picks the part has already picked it.


• As you just saw, we are activating “real” and “fake” part presents, on rung 6 we basically activate the Partpresent.Pres (which is the tag the PLC will use to know if it has a part on that station), based on the Tryout bit, if Tryout is activated (meaning we will run with no parts), then it will use the part present memory (the fake parts), and if the Tryout bit is not on (meaning we will run with real parts), then it will use the Partpresent.PresSwitches (the one mapped directly to the part presents connected on the I/O blocks). And the bottom branch checks both if the real part is not present and the fake part memory is not present, the that means the part is not present.


• These three rungs show the process of a robot loading parts. Rung 7 is used to reset the loading of a part every time the simulation is reseted.
• Rung 8: This rung shows us that when the Station is in Load Position “LoadPosn”, the robot R04 is at drop1 “CN150R04.Seg[30]” and at End of Segment “EndOfSeg” and the Part is Present “PresSwitches”. The Load Action Complete “Load.Comp” will be latched.
Another variation, instead of the Robot meeting the condition of the Drop and End Of Segment, is if it meets the Clear conditions “Clear.O” (meaning is at home or it has no part). The Load Action Complete “Load.Comp” will be latched as well.
• Rung 9: Lastly, when all the Resets “ResetProcComp” and the Part Not Present “NotPres” conditions are met, the Load Complete will be unlatched.

Clamps

• The routine describes what will the action do, in this case “S060_CloseClamp” means the clamp will be closed once the routine is completed, this branch is located right after the Part Check Clear of Robot R04, which helps us to interpret that once the Robot is clear the clamps start to work on the Station.


• Looking at the Hardwire Prints, as well as in the WinMOD file. We see that this station has 2 Clamps and 2 pin clamps. We are interested in the clamps; these are connected to Block 01 “CN160BK01”. Each one connected to a different Input.
Clamp_1 ---> .I_10 Clamp_1 ---> .I_11
Clamp_1 ---> .I_12 Clamp_1 ---> .I_13


• In the logic, we can see the rung where the mapping of the Clamp takes place. For example, BK01.I.D11 and C02AL.PX2. If the connections are made successfully on WinMOD, once the SIM is active “running” the Input (XIC) needs to be on (green), for the output C02AL.PX2 to take place (energize) as well. This will mean the clamp is in position. This tag comes directly from WinMOD, by seeing this we can determine if a connection is wrong on WinMOD or if a tag is wrong in the PLC.


• Rung 2: The same process as in rung 1 needs to take place, however, this time it is the Clamp from the right side. This is because these two clamps are connected in the same valve.


• Rung 3: If left and right Clamp 1 “C02AL.PX2 and C02AR.PX2” are in position, and Clamp 1 is NOT open “ClampOpen.Out”, then the OTE will be energized, and the action of Clamp Closed will take place “ClampClose.Comp”. Meaning the clamp has fully closed.


• Rung 4: When the Fast Stop is not activated “StopModes.Active.FastStop”, the Station has all Parts Present “Sts.AllParts” and all the process completes are not ON “MainCycle.ProcComp”. The clamp will get the condition to be able to run in Auto mode.


• To see where the Main Cycle comes from, we cross-reference it. Allowing us to see what process is taking place on the other side. When the cross-reference tab is open, press OTE, and this takes us to a new rung.




• Main Cycle Rung 2: This shows us that for the output of ”All Completes Are On” to take place, robots CN160R01, R02 and R03, which are welders. Need to be done welding and clear, so they meet their condition “.ProcComp”.


• Rung 5 shows us the clear conditions for this action, this means when the clamp will be clear of every other motion that could result in a collision. As you can see, it will be clear once every robot the station interacts with is at home or doesn’t carry a part.


• Rung 6 is basically telling us that the clamp will be allowed to have output power once the air it works on is ON and all the minor safeties are ok.


• Rung 7: This rung gives us two options, to run the process in Auto or Manual. From the top line, we interpret that once the Station is in cycle “InCycle”. To see this happening, we must cross reference it to see where it comes from. Besides this we expect the Auto condition of the clamp “.Auto” to be ON, the clamp close in NOT complete (which means the clamp is still opened), the clamp is clear “.Clear” and the Output Power “.OutPwr” is also met, then command output “.Cmd” will take place.
• Under this line, we see the manual option. If the station is in Manual Mode “Mode.Man”, the Push Button is requested “.PB”, the clamp needs to be clear “.Clear” and the Output Power “.OutPwr” is also on, then command output “.Cmd” will take place.


• Then in rung 8 we can see the “.CMD” is used to activate the “.OUT” condition. It just requires for the clamp not to have its “CMD” and “OUT” conditions ON in the open action. This means we are not trying to open and close the clamp at the same time.
• Rung 9: In this rung we are finally interacting with a physical output, this time if all the conditions were met and we can get the “ClampClose.Out” to be on, the output will energize sending a signal to the Pneumatic Manifold “PM1.O.VS14”. In the real world this will power a manifold, in our simulation this signal will make our valve move. If something is not working on your simulation, this right here could lead you to find the problem, maybe a connection is wrong on WinMOD (if you see this output on and nothing moving), or maybe this tag is wrong in the PLC and it is powering something else.



Tracing back common robot faults.

Part presents and clamps are some of the most common faults you might get while working on a simulation, now that you know how they work the other thing that causes a lot of issues are robots.
Robots have their own task where they have all the logic to make them work in a general way, but still each station the robot works in will have robot logic to be able to “utilize it”, this is to initialize the robot, make it pick and drop, or to make it apply seal, weld, or do any other process depending on the robot type. We will focus on the most common issues found during virtual commissioning.
• The robot stopped moving and you have no idea why.
o The first thing you need to look at is the Motion Enable bit (field “1”), this bit should be ON if all the conditions on the PLC are met for the robot to be able to move and is a good starting point if you have no idea why the robot is not moving.



The first thing you need to do is look for the station this robot works at, then look for the V100 Robot Common routines and open the one for the robot you are investigating. Look at the right side of the screen while scrolling (where the OTEs are located) and look for the Motion Enable bit. From there just cross reference whatever is not ON to determine the cause.
As you can see in that routine we can find almost all the things related with that robot and station CN160, from the initialization of the robot, how we set the style to the robot after it gets initiated, if it is in a valid segment, if it has motion, if it is clear and also the decision codes, this last part is really important because decision codes cause also a lot of problems, so have in mind where the decision codes are being send to the robot.
o Field “2” in the robot macro shows the decision code we are receiving from the PLC, in this example we are receiving DC 0, in Acptd codes it will display the accepted decision codes, so make sure you have the ones expected from the PLC and make sure the PLC is sending the right ones.
o Field “5” shows the current robot style and field “4” shows that the robot is at a RTC (request to continue) point, this means its at End of Cycle and waiting from instructions to continue moving forward.
o Finally, another issue you might have is the robot is not moving and there is no apparent way to tell what is happening, you might have a style and even motion, but there is still no motion. This might be because the robot is waiting on further instructions, to know if this is the case you can look into “O.PathSegContOK” (field “3”, if it is on then this is not the problem, but if it is not on it might be worth looking at it on the PLC).


We know we are looking for a direct output from the PLC to the robot. Go to the robot main task and inside look for the MapOutputs routine, look at the right side while scrolling down and look for an “O” (notice the difference between a 0 and a O, the zero will be used most likely in FLL functions while the O will be used to map the outputs to the robot). Right click it and monitor the tag (the “O”), expand the parent tag “O” and look for the child tag “O.PathSegContOK”, cross reference it and look for what the cause might be. It might be one or it might be several causes, whatever it is, it’s a good starting point to find what is causing the issue.
